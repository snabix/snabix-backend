# Changelog

Все важные изменения в этом проекте документируются в этом файле.

Формат основан на [Keep a Changelog](https://keepachangelog.com/en/1.1.0/),
а сам проект следует принципам [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [Unreleased]

### Fixed
- Добавление объявления в избранное больше не падает, если побочное уведомление владельцу не удалось доставить.
- DTO объявления теперь нормализует legacy `address_snapshot`, чтобы старые или неполные адреса не ломали загрузку личных объявлений и ответ после создания.

### Added
- Добавлен защищенный service API для Telegram-бота: health, service identity и базовая статистика.
- Добавлены API-действия для удаления одного уведомления и очистки всех уведомлений пользователя.
- Добавлена команда `shared:cleanup-storage` и ежедневный scheduler retention для `storage/logs`, generated API docs, временных uploads, media-library temp и debugbar-файлов.
- Добавлена безопасная команда `media:cleanup-orphans` для поиска и ручного удаления постоянных media-файлов без записей в таблице `media`.
- Добавлен комплект проектной документации по архитектуре, локальной разработке, релизам, секретам, тестированию, уведомлениям, lifecycle объявлений, lifecycle медиа и Telegram-боту.

## [0.6.30] - 2026-06-25

### Added
- В Filament-ресурсы журнала действий и ошибок добавлены кнопки быстрой очистки с confirmation-модалкой.
- Добавлен локальный Mailpit для перехвата SMTP-писем в development-окружении с веб-интерфейсом на порту `8025` и постоянным Docker volume для истории писем.
- Добавлена многоуровневая защита основной PostgreSQL от тестовых и destructive-команд: проверяется фактический Laravel config, а `migrate:fresh`, `migrate:refresh`, reset, rollback и `db:wipe` разрешены только для `db-test/snabix_test` в окружении `testing`.
- Для PostgreSQL-контейнеров включено журналирование DDL-команд, чтобы операции изменения схемы можно было однозначно отследить по Docker logs.
- Добавлена единая система пользовательских уведомлений с двумя каналами доставки: внутри сайта через database notifications и письмами через очередь `notifications`.
- Добавлены настройки подписок по типам событий, обязательный in-app канал для security-событий и API чтения/обновления preferences.
- Добавлены API списка уведомлений, unread counter, отметки одного уведомления и всей ленты прочитанными.
- Успешный вход в аккаунт создаёт security-уведомление и при включённой подписке отправляет email.
- Добавлены feature-тесты preferences, database-ленты, read-state и dispatch security-уведомления.
- Добавлен сброс настроек доставки текущего пользователя к значениям по умолчанию.

### Changed
- Форма характеристик категорий в Filament переведена на равную двухколоночную сетку секций.
- Форма создания и редактирования объявлений в Filament переведена на равную двухколоночную сетку секций.
- Публичный mapper новостных блоков теперь отдаёт только разрешённые поля для каждого `NewsPostBlockType`, а форма новостей показывает типизированные JSON-примеры.
- JSON-пример в форме новостей теперь вынесен в сворачиваемую секцию, чтобы не перегружать редактор контента.
- Форма новостей в Filament получила сетку `8/4` для основных данных и публикации, независимый полноширинный конструктор контента, красивый JSON-пример и code editor для данных блока.
- Инфопанель состояния системы в Filament переработана в более наглядный dashboard с общим индексом здоровья, сортировкой проблемных проверок и улучшенными карточками метрик.
- `RoleResource` Filament Shield перенесён в navigation group `Управление доступом`.
- Локальная почтовая конфигурация переключена с внешнего Mailtrap SMTP на контейнер `mailpit:1025` без внешней сетевой зависимости.
- Queue worker переведён на Redis cache и единственную объявленную очередь `notifications`, чтобы уникальные email jobs и проверка `queue:restart` не зависели от таблицы `cache` и не опрашивали отсутствующие RabbitMQ queues.
- Пустой context уведомления теперь сериализуется как JSON-объект, что сохраняет единый API-контракт с frontend.
- Повторная отправка email verification больше не инвалидирует предыдущий ещё действующий код: код повторно используется в зашифрованном виде, поэтому задержанные queue retries не доставляют уже недействительные значения.
- При вынужденной замене verification-кода предыдущий код остаётся валидным ещё 10 минут, чтобы SMTP retry и задержанная доставка не блокировали подтверждение email.

### Fixed
- API списка уведомлений теперь нормализует payload и не отдаёт лишние/legacy поля из `notifications.data`, чтобы frontend-контракт не падал на старых уведомлениях.
- In-app уведомления теперь создаются сразу для события входа и добавления чужого объявления в избранное, а email-канал остаётся очередным.

## [0.6.29] - 2026-06-06

### Added
- В таблицу категорий Filament добавлен фильтр для отображения только корневых категорий или только дочерних категорий.
- Добавлена загрузка иконки категории через Filament-форму создания и редактирования категории.
- Добавлена связь категории с медиахранилищем через полиморфную media-связь `media.model_type/model_id`.
- Публичные endpoints категорий снова возвращают `icon` с URL загруженной иконки.

### Changed
- Текущая иконка категории отображается в `FileUpload` после сохранения и заменяется только при загрузке нового файла.
- Переход `categories` и `listings` на UUID доведен до совместимого состояния в Filament, CLI и feature-тестах.
- Модели категорий и объявлений получили нативную Spatie media-интеграцию поверх UUID и полиморфных media-связей.
- `migrate:refresh` снова совместим с Postgres self-reference для `categories.parent_id` и со старыми окружениями, где миграция `icon_media_id` уже была записана в истории.
- Filament-форма категории переведена на `SpatieMediaLibraryFileUpload`, поэтому иконка теперь редактируется через media collection без временного `local` upload state.
- `User` и `News` переведены с ручных `media_id`/`cover_media_id` на полноценные Spatie media collections, чтобы все owner-модели работали через единый полиморфный media flow.
- Загрузка аватара пользователя, иконки категории и изображений объявления теперь использует нативный Spatie API (`addMedia` / `addMediaFromDisk`) вместо ручного сохранения файлов через временные записи.
- Структура путей медиа в storage стала человекочитаемой: файлы раскладываются по типу и коллекции (`images`, `documents`, `videos`, `files`) с UUID-директорией вместо числового `media_id`.
- Локальный `public/storage` symlink перепривязан на текущий проект, поэтому URL из `public` disk снова совпадают с реальным путём файлов.
- Таблица характеристик категорий в Filament теперь безопасно нормализует `options`, даже если компонент получает строку вместо массива.
- Frontend-контракты каталога и публичных объявлений синхронизированы с UUID-категориями backend, поэтому `/categories/list`, `/categories/{id}/branch` и `/public/listings` больше не разваливаются на client-side schema parsing.
- Frontend lockfile пересобран под Railpack-совместимый Node 22 / npm 10 flow; в `package.json` закреплены `engines`, а `package-lock.json` теперь явно содержит `@emnapi/core` и `@emnapi/runtime`, из-за отсутствия которых падал `npm ci` в production build.
- Списочная карточка объявления на frontend переработана под новый промо-лейаут: крупное изображение, верхние бейджи, акцентная цена, расширенный блок продавца и метаданные публикации в более витринном формате.
- В списочной карточке объявления восстановлена интерактивность кнопки избранного поверх overlay-ссылки карточки, а seller-секция теперь ведёт на отдельный frontend route `/sellers/[seller]` с рабочей заглушкой под будущую страницу продавца.
- Карточки объявлений получили такой же tilt-эффект, как карточки категорий, а детальная страница объявления вынесена в публичный маршрут `/listings/[id]` с backend endpoint `GET /api/v1/public/listings/{listingId}`, поэтому свои и чужие опубликованные объявления открываются единообразно вне кабинета.

### Removed
- Из схемы новостей убраны legacy-колонки `news_posts.cover_media_id` и `news_post_blocks.media_id`; связь с медиа теперь полностью полиморфная через таблицу `media`.
